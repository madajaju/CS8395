== Purpose
Explores the guarantees a message broker provides when moving data between systems and the tradeoffs of each approach.

== Outcomes
- Define "At-Most-Once", "At-Least-Once", and "Exactly-Once" delivery
- Recognize how network failures impact delivery guarantees
- Design systems that can handle duplicate messages (idempotency)

== Overview
When sending a message over a network, failure is inevitable. The "Delivery Semantics" of a system define what happens when things go wrong.

== 1. At-Most-Once (Fire and Forget)
The message is sent, but no acknowledgment is required.
- **Guarantee:** 0 or 1 delivery.
- **Pros:** Lowest latency, highest throughput.
- **Use Case:** High-volume telemetry (e.g., CPU metrics).

== 2. At-Least-Once (Acknowledged Delivery)
The producer retries sending until it receives an acknowledgment.
- **Guarantee:** 1 or more deliveries.
- **Pros:** No data loss.
- **Cons:** Consumers **must** handle duplicate messages.
- **Use Case:** Payments, order processing.

== 3. Exactly-Once
The system ensures the message is processed exactly once.
- **Guarantee:** Exactly 1 logical processing.
- **Reality:** Achieved via **At-Least-Once + Idempotency**.
- **Cons:** Significant performance overhead.

== Code Example: Go (At-Most-Once)
[source,go]
----
func sendMetric(metric string) {
    conn, _ := net.Dial("udp", "metrics:8125")
    fmt.Fprintf(conn, metric) // No check for receipt
}
----

== Code Example: Python (At-Least-Once)
[source,python]
----
def publish_with_retry(channel, message):
    while True:
        try:
            channel.basic_publish(
                exchange='',
                routing_key='task_queue',
                body=message,
                properties=pika.BasicProperties(delivery_mode=2)
            )
            break
        except pika.exceptions.AMQPError:
            time.sleep(1)
----

== Code Example: Java (Idempotent Consumer)
[source,java]
----
public void onMessage(OrderEvent event) {
    if (processedStore.exists(event.id)) {
        log.info("Duplicate detected, skipping: " + event.id);
        return;
    }
    process(event);
    processedStore.save(event.id);
}
----

== Risks and Tradeoffs
- **Ghost Messages:** In "At-Most-Once," data can be lost.
- **Write Amplification:** "At-Least-Once" increases system load.
- **Transactional Overhead:** True "Exactly-Once" requires complex coordination.
