== Purpose
Provides practical strategies for building services that can process the same message multiple times without unintended side effects.

== Outcomes
- Explain the necessity of idempotency in distributed systems
- Implement "Idempotency Keys" and "De-duplication Stores"
- Distinguish between natural and synthetic idempotency

== Overview
In a system with **At-Least-Once** delivery, consumers will receive duplicate messages. Idempotency is the property where an operation can be applied multiple times without changing the result.

== 1. Natural Idempotency
Some operations are inherently idempotent.
- **Example:** Setting a status to "ACTIVE".
- **Mathematical Analogy:** `f(x) = f(f(x))`

== 2. Synthetic Idempotency
Operations that are NOT naturally idempotent (e.g., "Withdraw $50") must be made idempotent.
- **Idempotency Key:** A unique ID generated by the producer.
- **De-duplication Store:** A fast store (Redis/SQL) to track processed keys.

== De-duplication Workflow
1. Consumer receives message with `Idempotency-Key`.
2. Check if key exists in de-duplication store.
3. If exists, skip processing (it's a duplicate).
4. If not, process message and save key atomically.

== Code Example: Node.js (Middleware)
[source,javascript]
----
async function idempotencyGuard(req, res, next) {
  const key = req.header('X-Idempotency-Key');
  const existing = await cache.get(key);
  if (existing) {
    return res.status(200).send(existing);
  }
  res.on('finish', () => cache.set(key, res.body));
  next();
}
----

== Code Example: Python (Handler)
[source,python]
----
def on_payment_received(msg):
    if db.payments.find_one({"idempotency_key": msg.id}):
        logger.info(f"Duplicate payment {msg.id} - ignoring.")
        return

    db.payments.insert_one({
        "idempotency_key": msg.id,
        "amount": msg.amount,
        "status": "processed"
    })
----

== Risks and Tradeoffs
- **Key Expiration:** Stores cannot grow forever (need TTL).
- **Transactional Integrity:** State and key must be updated together.
- **State Leakage:** Failures between business logic and key storage.
