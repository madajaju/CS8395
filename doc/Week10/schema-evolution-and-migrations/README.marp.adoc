== Purpose
Provides a framework for evolving database schemas in a distributed system without downtime.

== Outcomes
- Apply the "Expand and Contract" pattern to database changes.
- Identify "Breaking" vs "Non-breaking" schema changes.
- Implement automated migration pipelines that support safe rollbacks.

== Overview
In a distributed system, you cannot assume all instances of a service will upgrade at once.
- Schema changes must be decoupled from code changes.
- Backward compatibility is essential.

== Breaking vs. Non-breaking Changes
- **Non-breaking:** Adding a new optional column or table. Safe to apply while old code is running.
- **Breaking:** Renaming a column, changing a data type, deleting a column. Requires orchestration.

== The "Expand and Contract" Pattern
To perform a breaking change without downtime:
1. **Expand:** Add the *new* column to the database (nullable/optional).
2. **Migrate (Sync):** Update application to write to *both* but read from *old*.
3. **Migrate (Data):** Run background job to copy existing data.
4. **Switch:** Update application to read from the *new* column.
5. **Contract:** Remove the *old* column from the database.

== Code Example: SQL (Expanding a Table)
[source,sql]
----
-- V1__Add_New_Column.sql
ALTER TABLE orders ADD COLUMN status_v2 VARCHAR(50);
-- Safe to apply while V1 code is running
----

== Code Example: Python (Multi-Step Write Logic)
[source,python]
----
# Transition state: Writing to both old and new
def update_order_status(order_id, new_status):
    # Write to old column for backward compatibility
    # Write to new column for the future
    db.execute("UPDATE orders SET status = %s, status_v2 = %s WHERE id = %s", 
               (new_status, new_status, order_id))
----

== Code Example: Go (Checking Migration Version)
[source,go]
----
// Ensure the application only starts if the schema is compatible
func ensureSchemaVersion(db *sql.DB) error {
    version, _ := getCurrentSchemaVersion(db)
    if version < RequiredVersion {
        return fmt.Errorf("incompatible schema version: %d", version)
    }
    return nil
}
----

== Risks and Tradeoffs
- **Duration:** This pattern is slow and requires multiple deployments.
- **Data Consistency:** Syncing data can cause lock contention or race conditions.
- **Testing:** You must test all transitional states.
