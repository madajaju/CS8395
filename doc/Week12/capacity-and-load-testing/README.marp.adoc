== Purpose
Provides a framework for identifying the performance limits of a distributed system before production traffic reaches them.

== Outcomes
- Contrast Load Testing, Stress Testing, Soak Testing, and Spike Testing.
- Identify common bottlenecks such as CPU, memory, network, and database connection limits.
- Define SLIs and SLOs using test results instead of guesswork.

== Overview
How many users can your system handle, and where does it actually fail first?
- In distributed systems, the bottleneck is often not where teams expect.
- Capacity testing is about finding limits before production finds them for you.

== Core Testing Types
1. **Load Testing:** Expected traffic volume to verify normal performance targets.
2. **Stress Testing:** Push beyond expected load until the system breaks.
3. **Soak Testing:** Hold a steady load for a long period to expose leaks and exhaustion.
4. **Spike Testing:** Rapidly increase traffic to observe burst handling and autoscaling behavior.

== Golden Signals
- **Latency:** How long requests take.
- **Traffic:** How much demand hits the system.
- **Errors:** How often requests fail.
- **Saturation:** How close the service is to capacity.

== Code Example: Go (Custom Hammer)
[source,go]
----
func hammer(url string, workers int) {
    for i := 0; i < workers; i++ {
        go func() {
            for {
                http.Get(url)
            }
        }()
    }
}
----

== Risks and Tradeoffs
- **Cost:** Large-scale tests can consume significant compute and bandwidth.
- **Environment Parity:** Staging results rarely match production exactly.
- **Data Cleanup:** Tests can pollute storage with synthetic records.
- **Blast Radius:** Poorly isolated tests can affect neighboring infrastructure.
