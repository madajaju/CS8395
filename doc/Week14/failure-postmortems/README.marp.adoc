== Purpose
Provides a framework for analyzing system outages to maximize learning and avoid blame.

== Outcomes
- Define the "Blameless Postmortem" philosophy.
- Identify the difference between Proximate and Root causes.
- Implement a structured postmortem document.

== Overview
Failures in a distributed system are inevitable.
- High-performing teams handle failure as a clinical analysis.
- **Goal:** Prevent recurrence, not find a scapegoat.

== The Blameless Philosophy
Focus on the **system** and **processes**, not the individual.
- "If one person can take down the system, it's a system failure."
- Encourages honesty and transparency.

== Proximate vs. Root Cause
- **Proximate Cause:** The immediate event (e.g., "The disk was full").
- **Root Cause:** The fundamental issue (e.g., "Monitoring failed to alert at 90% and no automated cleanup existed").
- **Technique:** Use the "Five Whys."

== Postmortem Structure
| Section | Content |
| :--- | :--- |
| **Summary** | Overview and impact. |
| **Timeline** | Detection, response, resolution. |
| **Root Cause** | Deep architectural/process failure. |
| **Action Items** | Concrete tasks to prevent recurrence. |
| **Lessons** | What went well vs. wrong? |

== Code Example: Python (Data Collection)
[source,python]
----
# Script to capture system state after a failure
def capture_diagnostics():
    logs = cloudwatch.get_logs(last_30_mins=True)
    metrics = prometheus.query("rate(http_errors[30m])")
    with open("failure_context.json", "w") as f:
        json.dump({"logs": logs, "metrics": metrics}, f)
----

== Risks and Tradeoffs
- **Time Sink:** Writing good postmortems is time-consuming.
- **Fear:** If not truly blameless, people hide mistakes.
- **Action Item Drift:** "New feature work" often overrides fixes.
