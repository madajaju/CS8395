== Purpose
Master the skill of articulating architectural decisions by focusing on the "Why" and the "Instead Of."

== Outcomes
- Construct a Decision Matrix for architectural choices.
- Articulate the "Cost" of the chosen architecture (losing options).
- Identify the long-term maintenance impact of a design.

== Overview
Senior architecture isn't about the "perfect" solution; it's about the "least bad" solution for the constraints.
- A Tradeoff Narrative explains Path A vs. Path B.
- What was sacrificed to get there?

== The Core Elements
1. **The Context:** Problem, Constraints (Time, Cost, Performance, Skillset).
2. **The Options:** Alternatives considered and why.
3. **The Decision:** Chosen path and deciding factor (e.g., "Availability > Accuracy").

== The Decision Matrix
| Option | Scalability | Complexity | Cost | Decision |
| :--- | :--- | :--- | :--- | :--- |
| **Monolith** | Low | Low | Low | No (Needs scale) |
| **Microservices** | High | High | High | **Yes (Chosen)** |
| **Serverless** | High | Medium | Medium | Maybe (Cold Starts) |

== Code Example: ADR (Markdown)
[source,markdown]
----
# ADR 001: Choosing Sagas over 2PC
## Context: 
Order processing spans 3 services.
## Decision: 
Use an Orchestrated Saga.
## Consequences:
- We must handle eventual consistency in the UI.
- No more database-level JOINS.
----

== Code Example: Python (Documenting Hacks)
[source,python]
----
def process_data_slowly():
    # Tradeoff: Synchronous call to simplify first release.
    # Sacrificing throughput (max 10 rps) to reduce complexity.
    # Will migrate to message queue in Phase 2.
    call_blocking_api()
----

== Risks and Tradeoffs
- **Analysis Paralysis:** Documenting instead of building.
- **Decision Fatigue:** Focus on "One-Way Door" decisions.
- **Stale ADRs:** Outdated documentation becomes a liability.
